# Internationalization (i18n)

## Use This File When

- Adding, renaming, splitting, regenerating, or validating translation keys
- Changing `next-intl` usage, locale routing, message loading, or cache-safe translation access
- Touching `messages/`, `public/messages/`, `src/i18n/`, or `src/lib/i18n/`

## Do Not Use This File For

- Page prose, FAQ content, or SEO copy ownership; use `content.md`
- Generic Next.js routing/layout behavior; use `conventions.md` and `node_modules/next/dist/docs/`
- Generic copywriting or brand messaging decisions

## API Decision Table

| Need | Use |
|------|-----|
| Server Component text | `getTranslations()` |
| Metadata text with known locale | `getTranslations({ locale, namespace })` |
| Client Component text | `useTranslations()` |
| Rich text interpolation | `t.rich()` |
| Cache Component-safe translation access | explicit `locale` argument |
| Locale-aware static rendering | `setRequestLocale(locale)` in `[locale]/layout.tsx` |
| Full runtime message catalog | `loadCompleteMessages(locale)` only when truly required |

## Framework

- **Library**: next-intl
- **Locales**: `en`, `zh` (default: `en`)

## Translation Files

```
messages/
├── en/critical.json    # Above-the-fold, SEO-critical
├── en/deferred.json    # Below-the-fold, lazy-loaded
├── zh/critical.json
├── zh/deferred.json
└── {en,zh}.json        # Compatibility artifacts for tests/scripts (not runtime truth)
```

### Canonical Source Rule

- Runtime canonical source is **split files only**: `messages/{locale}/{critical,deferred}.json`
- `messages/en.json` / `messages/zh.json` are compatibility artifacts for tests/scripts
- Runtime code must **not** depend on flat root files as a fallback source
- Current repo truth does **not** include a live `src/sites/**/messages/**` runtime overlay layout. Do not write code that assumes that path exists unless the structure is introduced in the same branch and this rule is updated with it.

### Runtime Loading Rule

- `src/lib/load-messages.ts` and `src/i18n/request.ts` must share the same runtime truth source
- **Never** use self-HTTP fetch to load the app's own translation files on the server runtime path
- Server runtime fallback must reuse split-source aggregation, not `messages/{locale}.json`
- Translation runtime truth is currently the shared split bundles under `messages/{locale}/`
- Root layout must emit the correct server-rendered `<html lang={locale}>`; client-side patching is not a valid substitute.

### Quartet Discipline

Treat split source, flat compatibility files, and copied public bundles as one translation family:

- canonical source: `messages/{locale}/critical.json` + `messages/{locale}/deferred.json`
- compatibility view: `messages/{locale}.json`
- runtime copy: `public/messages/{locale}/critical.json` + `public/messages/{locale}/deferred.json`

Rules:

- If split source changes, inspect flat and public copies
- If only flat files change, regenerate them instead of editing by hand
- Use `pnpm review:translation-quartet` for split/flat/public parity

## Using Translations

### Server Components
```typescript
import { getTranslations } from 'next-intl/server';

const t = await getTranslations('products');
return <h1>{t('pageTitle')}</h1>;

// With explicit locale (for generateMetadata)
const t = await getTranslations({ locale, namespace: 'products' });
```

### Client Components
```typescript
'use client';
import { useTranslations } from 'next-intl';

const t = useTranslations('products.card');
return <span>{t('moq')}</span>;
```

### Rich Text
```typescript
t.rich('message', { bold: (chunks) => <strong>{chunks}</strong> });
t.has('someKey'); // Check existence
```

## Key Naming Convention

### Format

- **camelCase** for all key segments: `pageTitle`, `submitSuccess`, `firstNamePlaceholder`
- **Namespace keys cannot contain `.`**

### Structure: `namespace.section.key`

Nesting depth: **2-4 levels**. Avoid exceeding 4 levels.

```
✅ catalog.overview.title              (3 levels)
✅ contact.form.firstName              (3 levels)
✅ home.hero.cta.primary               (4 levels — max)
❌ common.toast.form.contact.success   (5 levels — too deep, flatten)
```

### Patterns

| Scenario | Pattern | Example |
|----------|---------|---------|
| Page section | `page.section.key` | `home.hero.title` |
| Form field | `namespace.form.fieldName` | `contact.form.firstName` |
| Form field placeholder | `namespace.form.fieldNamePlaceholder` | `contact.form.firstNamePlaceholder` |
| CTA buttons | `namespace.section.cta.variant` | `home.hero.cta.primary` |
| Ordered items | `namespace.section.stepN.key` | `home.chain.step1.title` |
| Metadata/SEO | `metadata.pageName.title/description` | `metadata.products.title` |
| Error messages | `errors.domain.key` | `errors.products.title` |
| Common/shared | `common.key` | `common.loading`, `common.close` |

### Rules

- Descriptive keys over numbered keys when items have distinct semantics
- Suffix pattern for related variants: `firstName`, `firstNamePlaceholder` (same level, not nested)
- `title` / `description` / `cta` are standard sub-keys for any section
- New namespaces must match an existing page or domain (`home`, `catalog`, `contact`, `common`, `errors`, etc.)

## Adding Translations

1. Add keys to both `en/` and `zh/` split files
2. Keep structure identical across locales
3. Follow key naming convention above
4. Run `pnpm i18n:full` (scan + sync + validate + slug-check)
5. Flat compatibility files are regenerated from split files by repo tooling
6. If split files changed, ensure regenerated flat files are included in the commit (pre-commit now re-stages them)

## Static Rendering / Cache-Friendly Usage

### Required: `setRequestLocale`

next-intl requires `setRequestLocale(locale)` before calling next-intl hooks in locale-based static rendering. In this repo, keep that call in `[locale]/layout.tsx`:

```typescript
import { setRequestLocale } from 'next-intl/server';

export default async function LocaleLayout({ children, params }) {
  const { locale } = await params;
  setRequestLocale(locale); // enables locale-aware static rendering
  // ...
}
```

### Forbidden in `"use cache"`

In cached segments, do not use request-bound next-intl helpers. Pass an explicit locale instead:

```typescript
// ❌ Error: reads request context
async function getCachedContent() {
  'use cache';
  const t = await getTranslations('home'); // Will fail!
}

// ✅ Correct: explicit locale parameter
async function getCachedContent(locale: 'en' | 'zh') {
  'use cache';
  cacheLife('days');
  const t = await getTranslations({ locale, namespace: 'home' });
}
```

### Safe APIs for `"use cache"`

| API | Safe | Notes |
|-----|------|-------|
| `getTranslations({ locale, namespace })` | ✅ | Explicit locale |
| `loadCriticalMessages(locale)` | ✅ | Explicit locale |
| `loadCompleteMessages(locale)` | ✅ | Explicit locale; use only when full catalog needed |
| `getTranslations()` (no args) | ❌ | Reads request |
| `getLocale()` | ❌ | Reads request |

**Note**: `loadCompleteMessages(locale)` merges the entire catalog (`critical` + `deferred`).
It is safe with an explicit locale, but it should not be the default API for `"use cache"` segments.
Prefer `getTranslations({ locale, namespace })` or `loadCriticalMessages(locale)`;
only use the full-catalog helper when the complete catalog is truly required.

## Critical vs Deferred Translation Split

- `critical.json`: Text visible on first screen without scrolling or interaction
- `deferred.json`: Text that appears after scroll or interaction

This is an optimization guideline, not a hard boundary. If post-migration volume is
small enough that the split adds more complexity than value, consolidate.

After content truth unification, translation files contain only Layer 4 UI chrome —
not page content, not FAQ, not SEO metadata.
