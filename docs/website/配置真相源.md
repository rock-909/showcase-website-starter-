# 配置真相源

这个 starter 有两层配置面：

1. 当前网站真正读取的主配置：改这里，才是在改实际网站。
2. 给新项目替换时参考的辅助配置，外加少量兼容用的镜像配置：它们帮助理解替换面，但通常不是实际网站的主开关。

不要把两层混成同一个概念。

## 当前网站真正读取的主配置

| 范围 | 主配置 | 说明 |
| --- | --- | --- |
| 单站品牌、联系方式、社交链接、默认 SEO、公司事实 | `src/config/single-site.ts` | 当前单站主真相。 |
| sitemap、robots、lastmod、允许进入 sitemap 的公开页面清单 | `src/config/single-site-seo.ts` | SEO 公开页面和搜索引擎抓取规则的主配置。 |
| 导航 href | `src/config/single-site-links.ts` 和 `src/config/single-site-navigation.ts` | 页面链接从主路径配置派生。 |
| 页面表达输入 | `src/config/single-site-page-expression.ts` | 首页表达、产品/市场按钮/行动入口、custom project support 等页面输入面。 |
| 产品市场和 catalog truth | `src/config/single-site-product-catalog.ts` 与 `src/constants/product-specs/**` | 产品事实不能只改一个产品卡片文件。 |
| 运行时环境变量 | `src/lib/env.ts` | 网站运行时读取的 server/client 环境变量配置。 |

## 给新项目替换时参考的辅助配置和兼容用的镜像配置

| 文件 | 角色 | 规则 |
| --- | --- | --- |
| `src/config/website/profile.ts` | 兼容用的镜像配置 | 可以帮助派生项目理解替换面，但不能覆盖 `src/config/single-site.ts`。 |
| `src/config/website/seo.ts` | 兼容用的镜像配置 | 默认标题、标题模板、描述和 OG image 应从当前网站主配置派生；`siteUrl` 是静态 starter 替换占位，来自 `websiteProfile.domain`，刻意不跟运行时 `baseUrl` 环境覆盖同步。 |
| `src/config/website/navigation.ts` | 给新项目替换时参考的辅助配置 | 可说明导航结构，但实际运行导航应以 single-site route href 为准。 |
| `src/config/website/contact.ts` | 给新项目替换时参考的辅助配置 | 可说明接收方配置，但真实表单流向还要看运行时环境变量、API route 和 lead pipeline。 |
| `src/config/website/homepage.ts` | 给新项目替换时参考的辅助配置 | 当前只是替换面说明；运行时页面组合和按钮/行动入口主要看 `src/config/single-site-page-expression.ts`。 |
| `src/config/website/products.ts` | 给新项目替换时参考的辅助配置 | 只代表入口卡片，不代表完整产品事实。 |

## 同步规则

这些值不应无理由漂移：

- 公司名
- 站点名
- 联系邮箱
- 主社交链接
- SEO title template 的品牌部分

这些值可以在不同层表达不同角度，但必须有说明：

- 默认域名和 base URL：`src/config/website/seo.ts` 的 `siteUrl` 是给派生项目看的静态替换占位；运行时真实 base URL 以 `src/config/single-site.ts` 和 `src/lib/env.ts` 的环境覆盖为准。
- 电话格式：当前网站真正读取的主配置可以保留 starter placeholder，兼容用的镜像配置可以展示替换示例。
- founded year / established year：如果只是 demo placeholder，不能被当作真实业务事实。
- 页面表达：`src/config/single-site-page-expression.ts` 是运行时输入面，覆盖首页表达、产品/市场按钮/行动入口、custom project support 等页面输入。
- 产品卡片：`src/config/website/products.ts` 不是完整产品 catalog，不能只改这个文件就当作产品事实已改完。

## 派生项目替换顺序

1. 先改 `src/config/single-site.ts`。
2. 品牌、域名、SEO 或搜索引擎抓取变更时，检查 `src/config/single-site-seo.ts`。
3. 导航变更时，检查 `src/config/single-site-navigation.ts` 和 `src/config/single-site-links.ts`。
4. 页面表达、首页顺序、产品/市场按钮/行动入口或 custom project support 变更时，检查 `src/config/single-site-page-expression.ts`。
5. 产品事实变更时，检查 product catalog 和 `src/constants/product-specs/**`，不要只改 `src/config/website/products.ts`。
6. 再同步 `src/config/website/*` 这类辅助配置和兼容用的镜像配置；它们主要用于派生项目理解替换面，不是实际网站的主开关。
7. 再替换 `content/pages/{locale}/*.mdx`。
8. 最后检查 `messages/{locale}/*.json` 和 `public/images/**`。

## 后续治理规则

- 如果兼容用的镜像配置继续保留，应增加测试说明哪些字段必须同步。
- 如果兼容用的镜像配置不再带来价值，应在单独计划里逐步退役或改成从主配置派生。
- 不要在 Phase 1 引入新的 `defineSiteDefinition(...)`。只有确认人工同步继续造成实际漂移时，才进入结构合并。
